# React Native & Google Maps: Native Build Debugging Log

**Objective:** Implement an interactive Google Map (`react-native-maps`) using a personal API key in a React Native Expo project on a Windows machine.

* * *

## Phase 1: The "Blank Beige" Map Screen

-   **The Problem:** The app successfully compiled in standard Expo Go, the coordinates were correct, and the component mounted perfectly. However, the screen displayed a blank beige grid with only the Google logo at the bottom left.
    
-   **The Root Cause:** The standard Expo Go app comes pre-packaged with a shared Google Maps API key that frequently hits its daily quota. Because Expo Go cannot read a personal API key injected via `app.json` at runtime, the Google servers silently refused to send the map tiles. Furthermore, forcing `provider="google"` requires explicit native authorization.
    
-   **The Solution:**
    
    1.  Transitioned from the standard Expo Go app to a **Custom Development Build** (`npx expo run:android`).
        
    2.  Generated a personal, unrestricted Google Maps API key in the Google Cloud Console.
        
    3.  Injected the key into the native Android manifest by updating the `android.config.googleMaps.apiKey` in `app.config.js`.
        

* * *

## Phase 2: The Gradle `foojay` & Java 24 Crash

-   **The Problem:** Running the native build command resulted in an immediate crash: `Could not initialize class... NoSuchFieldError: IBM_SEMERU`.
    
-   **The Root Cause:** The PC was running **Java 24**, but React Native and the Android build tools strictly require **Java 17**. Because Java 24 is entirely new, it removed internal code that the Gradle `foojay` toolchain plugin relies upon, causing a fatal incompatibility. Additionally, the project generated an experimental Gradle 9.0 version.
    
-   **The Solution:**
    
    1.  Uninstalled Java 24 from the Windows system.
        
    2.  Installed JDK 17 (Java 17).
        
    3.  Updated the Windows `JAVA_HOME` environment variable to point to the new JDK 17 path.
        
    4.  Downgraded the Gradle wrapper from `9.0.0` to `8.6` inside `android/gradle/wrapper/gradle-wrapper.properties`.
        
    5.  Forced the project to use the stable `0.8.0` version of the foojay plugin by manually adding it to `android/settings.gradle`.
        

* * *

## Phase 3: The Locked File Timeout

-   **The Problem:** The build failed with: `Timeout of 120000 reached waiting for exclusive access to file: ...\gradle-8.6-all.zip`.
    
-   **The Root Cause:** Previous failed build attempts (due to the Java 24 crash) left "ghost" Java processes running in the background. These invisible processes locked the partially downloaded Gradle zip file, preventing the new, clean build from accessing or downloading it.
    
-   **The Solution:**
    
    1.  Forcibly killed all frozen background processes using the terminal command: `taskkill /f /im java.exe`.
        
    2.  Navigated to the corrupted cache folder (`C:\Users\thinu\.gradle\wrapper\dists\`) and manually deleted the locked `gradle-8.6-all` folder to force a clean download.
        

* * *

## Phase 4: Missing Android SDK Location

-   **The Problem:** The build ran successfully for over 2 minutes but failed at the end with: `SDK location not found.`
    
-   **The Root Cause:** The Gradle build system needs access to the core Android SDK tools installed by Android Studio to compile the final `.apk`. However, the project's native folder lacked the configuration file telling Gradle where to look on the Windows `C:` drive.
    
-   **The Solution:**
    
    -   Created/updated the `local.properties` file inside the `android` folder and explicitly defined the path: `sdk.dir=C:/Users/thinu/AppData/Local/Android/Sdk`
        

* * *

## Phase 5: The Windows 260-Character Path Limit

-   **The Problem:** During the final C++ compilation phase, the Ninja build system crashed with: `Filename longer than 260 characters`.
    
-   **The Root Cause:** The Windows operating system has a legacy limitation that prevents programs from creating or reading files if the full path length exceeds 260 characters. Because the project was nested deeply inside `OneDrive\Documents\GitHub\react_native\Native feature App`, the heavily nested C++ files generated by React Native exceeded this limit, causing Windows to kill the process.
    
-   **The Solution:**
    
    -   Moved the entire React Native project out of the deeply nested OneDrive folder to a much shorter, simpler path near the root directory (e.g., `C:\dev\NativeFeatureApp`). This reduced the string length of the file paths, allowing the compiler to successfully finish generating the native code.
        

* * *

**Result:** A fully functional, production-ready Custom Development Build running flawlessly on the Android emulator with a fully interactive map!
